这篇记录来自一次很典型的站内搜索优化。问题不复杂，但影响明显：用户能搜到内容，却觉得“慢”和“不准”。

## 现象

- 首次打开搜索弹层时有明显等待
- 结果摘要截取生硬，命中词不突出
- 输入两个关键词后，结果数量下降过快

## 处理思路

我们没有引入额外服务，继续沿用本地 `search.xml`。优化重点只有三件事：

1. 搜索弹层真正打开时再加载索引文件
2. 输入时只展示更短的摘要片段，减少认知负担
3. 给命中词高亮，降低用户“没搜到”的错觉

## 验收口径

| 指标 | 调整前 | 调整后 |
| --- | --- | --- |
| 搜索弹层首次可用时间 | 约 2.3s | 约 1.2s |
| 结果摘要长度 | 不固定 | 控制在 100 字左右 |
| 常见关键词命中率 | 一般 | 稳定 |

## 实施备注

有个细节很关键：不要在首页初始渲染时就把搜索数据全部拉下来。演示站内容少时不明显，换成真实站点后，差异会被放大。

```js
function shouldLoadSearchIndex(isOpen, loaded) {
  return isOpen && !loaded
}
```

## 结论

搜索不是做得越复杂越好。对这类静态站来说，优先把加载时机、关键词高亮和摘要截取做扎实，体验提升通常就已经足够明显。
