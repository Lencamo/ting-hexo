登录、短信发送、导出报表，这三类接口最容易在流量抖动时出问题。不是业务逻辑本身复杂，而是它们都天然需要限流。

## 常见误区

- 只在网关层做限流，不看业务接口差异
- 全部使用同一阈值，忽略高风险接口和普通接口的区别
- 返回“系统繁忙”，但没有记录真实触发原因

## 一个够用的分层方案

1. 网关层挡住异常突刺流量
2. 业务层按用户、IP 或租户做二次限制
3. 日志层明确写出命中的限流规则

## 计算思路

如果采用令牌桶模型，可以把一段时间内的可处理请求估算为：

$$
allowedRequests = rate \times window + burst
$$

其中 `rate` 是稳定放行速率，`burst` 是允许的瞬时峰值。

## 实践建议

```ts
function shouldThrottle(count: number, limit: number) {
  return count >= limit
}
```

示例代码很简单，但真正要做好的部分是观察口径：

- 限流是否误伤正常用户
- 命中后是否能快速定位来源
- 是否和安全策略、验证码策略一起工作

## 结论

限流不是为了“拒绝更多请求”，而是为了保证系统在异常时还能稳定提供核心能力。阈值可以慢慢调，但埋点和告警最好一开始就补上。
